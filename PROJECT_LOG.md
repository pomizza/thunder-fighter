## 🏗️ 关键架构决策

### 1. 零依赖原则 ⭐⭐⭐

**决策**：完全使用 vanilla JavaScript，不引入任何 npm 包。

**理由**：
- 游戏核心是 Canvas 绘制 + 状态机，原生 API 已够用
- 避免 React/Vue 等框架的运行时开销（游戏需 60 FPS）
- 减少供应链攻击面
- 教学价值：让读者看到"纯 JS 能写什么"
- 部署极简：纯静态文件

**代价**：
- 无现成库（动画/状态管理/工具函数）—— 全部自己写
- 测试基础设施需要自己搭（`vm.createContext` + `(0, eval)` hack）

**结果**：✅ 项目 364 KB Git 仓库大小，100% 可读

### 2. 模块拆分（方案 B）⭐⭐⭐

**问题**：游戏循环到第 20 轮时，`game.js` 涨到 937 行，圈复杂度 164。

**方案对比**：
| 方案 | 做法 | 优劣 |
|------|------|------|
| **A** | 改源码暴露 class 给测试 | 污染 6 个文件 |
| **B** | 改测试 runner hack 取 class | 0 生产代码改动 ✅ |

**最终选 B**：
- 抽出 `config.js`（149 行）— 常量集中
- 抽出 `gameInput.js`（109 行）— 键盘事件
- 抽出 `gameLogic.js`（217 行）— update 逻辑
- `game.js` 瘦到 706 行（-24%）

**关键 hack**：
```js
// ES6 class 不自动挂到 global
// 用 (0, eval) 间接 eval 取词法引用
const expr = classFiles.has(f)
  ? `(0, eval)("${exportName}")`  // 沙箱内 eval
  : exportName;                    // 普通 var/const
modules[exportName] = vm.runInContext(expr, ctx);
```

**经验**：`方案 B`（hack 测试基础设施）优于 `方案 A`（污染生产代码），前提是 hack 可控。

### 3. 状态机设计 ⭐⭐

**核心状态**（共 12 个）：
```
menu → selectShip → levelIntro → playing → levelClear → [Shop] → levelIntro → ...
                            ↘ gameover / victory ← (lives<0 / 终关通关)
```

**教训**：
- 第 18 轮发现 `Shop` 状态机漏洞（玩家死亡时 Shop 仍打开）
- 第 19 轮发现 `Boss 死亡` + `setTimeout` 竞态
- **状态机必须完全闭环**：每个状态都有明确的入口/出口/守卫

**4 重状态守卫**（最终设计）：
```js
if (this.state === 'levelClear' && this.player && this.player.alive
    && this.player.lives >= 0 && this.state !== 'gameover') {
  Shop.open(...);
} else if (this.state === 'levelClear') {
  this.gameOver();  // 玩家死亡兜底
}
```

### 4. 持久化策略 ⭐

**存什么**：
- `thunder_hi`：最高分
- `thunder_diff`：难度
- `thunder_ach`：成就解锁

**不存什么**：
- ❌ Replay 数据（太大、刷新即失）
- ❌ 玩家进度（重玩更有意思）
- ❌ 任何 PII

**理由**：localStorage 5 KB 上限，错误处理 try-catch（隐私模式可能抛错）

### 5. 测试基础设施 ⭐⭐

**挑战**：`vm.createContext` 加载的 ES6 class **不会自动挂到 global**。

**解决**：
1. `(0, eval)("ClassName")` 间接 eval（取词法引用）
2. `Object.defineProperty(ctx, 'globalThis', { get: () => ctx })` 模拟 globalThis
3. `ctx.performance = require('perf_hooks').performance` 用真实 performance

**结果**：43/43 测试通过，0 依赖，~50ms 跑完

**对比 c8/nyc**：
- **c8** 失败：V8 覆盖率 API 不跟踪 vm 沙箱（0%）
- **nyc** 需 babel 插件（破坏 0 依赖）
- **自制 coverage.cjs**：输出文件/类/模块统计，零依赖

## 🐛 重大 Bug 修复历史

### 第 3 关敌机消失（用户反馈）

**症状**：第 3 关无任何敌机出现。

**根因**：`levels.js` 中 `{ type: 'kamikaze', count: 12 }` **漏了 `enemyType` 字段**。`buildSpawnQueue` 读 `w.enemyType` 得到 `undefined`，`new Enemy(_, _, undefined)` 走默认 case，敌机属性不对但**活着**飞出屏外。

**修复**：`sed` 批量给 7 处加 `enemyType: 'kamikaze'`。

**教训**：data-driven 配置必须有**必填字段验证**。未来应在 `buildSpawnQueue` 加 `if (!type) throw`。

### 神风机两阶段 AI（第 3 轮）

**症状**：神风机速度 200 px/s，玩家来不及反应就飞出屏。

**修复**：分两阶段
- **入轨**（y < 100）：90 px/s 慢速，给玩家反应时间
- **攻击**（y >= 100）：220 px/s 锁定玩家

**教训**：所有"快速移动"敌机都应有"进入战斗距离"的过渡。

### Replay 画面不一致（第 16 轮 → 第 17 轮）

**症状**：第 16 轮实现 Replay 时，**回放画面 ≠ 录像时画面**。

**根因**：回放时，敌机 AI 真实运行（不是录像），所以**敌机移动不一致**。

**修复**：
- `applySnapshot(game)` 把录像状态同步到 game 对象
- 回放期间**跳过所有游戏逻辑**（只跑 Player 移动/射击）
- 敌机/敌弹/Boss 完全由录像控制

**教训**：录像系统 = **状态机**（不只是输入录制），必须能精确还原。

## 📐 设计权衡

### 自机选择（3 架战机）

| 战机 | 设计思路 | 影响 |
|------|----------|------|
| **蓝隼**（默认）| 平衡型 | 大多数玩家选 |
| **赤焰**（速度）| 12.5% 更快 + 双发 | 速度型爱好者 |
| **金星**（防御）| 6 HP + 8s 起步护盾 | 新手友好 |

**取舍**：
- 差异不能太大（避免"唯一最优解"）
- 差异必须有意义（避免"完全一样"）
- 数值差异在 ±20% 内（保留操作空间）

### 难度选择（3 档）

| 参数 | EASY | NORMAL | HARD |
|------|------|--------|------|
| HP | 6 | 5 | 4 |
| Bombs | 4 | 3 | 2 |
| 敌机 HP | ×0.7 | ×1.0 | ×1.3 |
| 敌机数量 | ×0.85 | ×1.0 | ×1.15 |
| Boss HP | ×0.8 | ×1.0 | ×1.3 |
| 连击窗口 | 2.0s | 1.5s | 1.2s |
| 震屏 | ×0.7 | ×1.0 | ×1.3 |
| 分数倍率 | ×0.8 | ×1.0 | ×1.5 |

**设计原则**：
- **不是改敌机速度**（那只是"反应时间"差异）
- 改 **HP/数量/连击窗口**（这些影响策略选择）
- 分数倍率让 HARD 模式"高分挑战"成为目标

## 📚 学到的经验教训

### 1. 单作者迭代的优缺点

**优点**：
- 决策快（无团队会议）
- 代码风格一致
- 可大胆 hack

**缺点**：
- 容易"小作坊"思维
- 缺乏 code review
- 测试覆盖可能偷懒

**应对**：用 CI（GitHub Actions）+ 自动化测试 + 静态分析（lint）

### 2. ES6 class 的隐藏陷阱

- `class X {}` **不挂到 globalThis**（与 var/const 不同）
- `class X {}` 是 **block-scoped**（不是 function-scoped）
- 测试时需要 hack（`vm.runInContext` + 间接 eval）

### 3. Canvas 游戏的性能优化

- **draw 调用次数**是瓶颈（每帧 < 100 次）
- **粒子/对象数组**必须有硬上限（防止内存爆炸）
- **离屏剔除**比"早期 draw"更有效
- **drawImage** 比 fillRect 快 5-10x（精灵图）

### 4. 状态机的可维护性

- **每个状态都有明确入口/出口**
- **状态守卫**而非"全局 if 链"
- **禁止"隐式状态转移"**（如 setTimeout 里的状态变化）

### 5. 测试的投入产出比

| 类型 | 工作量 | 价值 |
|------|--------|------|
| **核心逻辑单元测试** | 🟢 高 | 🟢 高 |
| **集成测试** | 🟡 中 | 🟢 高 |
| **UI/draw 测试** | 🔴 高 | 🟡 中 |
| **覆盖率统计** | 🟢 低 | 🟡 中 |

**本项目**：核心逻辑 + 集成 = 43/43 测试，跳过 UI（手动测）
